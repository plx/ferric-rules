# Data Structure Considerations

## Scope

This note focuses on low-risk, mostly drop-in data structure substitutions
inside `ferric-core` and `ferric-runtime`.

The goal is not to change algorithms or overall representation. The goal is
to identify places where we can:

- remove avoidable hashing work,
- reduce per-entry memory overhead,
- avoid tiny heap allocations, or
- use storage keyed more directly by the IDs the runtime already has.

Each item below should be treated as an independent experiment and measured on
its own. The right success bar is "unambiguous win on benchmark/runtime
behavior with no semantic regression," not "looks theoretically faster."

## Dependency Notes

Several of the candidates below would benefit from adding one or more small
dependencies:

- `rustc-hash`
  - Provides `FxHashMap` / `FxHashSet`.
  - Fast, deterministic, and usually a strong fit for internal trusted keys
    like `u32` IDs, symbols, and module-local names.
  - Best fit when DOS-resistant hashing is not a requirement.

- `ahash`
  - Provides `AHashMap` / `AHashSet`.
  - Also much faster than the default hasher, but randomized.
  - Better fit if we want a faster default without fully deterministic hashes.

- `compact_str` (optional, lower priority)
  - Useful if we want to optimize long-lived short strings without changing
    surrounding APIs much.
  - Secondary to simpler wins like `Box<str>`.

`hashbrown` by itself is not an obvious performance lever here: Rust's
standard `HashMap`/`HashSet` already use the same underlying table
implementation. In this codebase, the more meaningful change is usually the
hasher or the key layout, not switching from `std` to `hashbrown`.

## Evaluation Guidance

For each experiment:

1. Apply one change only.
2. Run the existing benchmark suite (`waltz`, `manners`, and any targeted
   microbenchmarks added for the change).
3. Capture memory impact separately when possible.
4. Keep the public API and semantics unchanged.
5. Revert immediately if the benefit is small or workload-dependent.

## Candidates

### 1. Use a Faster Hash Backend for Internal ID and Symbol Tables

**Current structures**

- Many hot-path maps and sets in `ferric-core` and `ferric-runtime` use
  `std::collections::HashMap` / `HashSet`.
- Representative sites include:
  - `crates/ferric-core/src/fact.rs`
  - `crates/ferric-core/src/alpha.rs`
  - `crates/ferric-core/src/token.rs`
  - `crates/ferric-core/src/negative.rs`
  - `crates/ferric-core/src/exists.rs`
  - `crates/ferric-core/src/ncc.rs`
  - `crates/ferric-runtime/src/engine.rs`
  - `crates/ferric-runtime/src/functions.rs`

**Proposed substitution**

- Introduce a crate-wide alias layer for internal maps/sets, backed by either:
  - `rustc_hash::FxHashMap` / `FxHashSet`, or
  - `ahash::AHashMap` / `AHashSet`.

**Where to use it first**

- Integer-keyed or symbol-keyed tables.
- Sets storing `FactId`, `TokenId`, `ActivationId`, `RuleId`, `SlotIndex`, and
  similar small internal key types.
- String-keyed runtime registries once their key layout is stabilized (see the
  name-registry items below).

**Additional adjustments**

- Add type aliases such as `FastMap<K, V>` and `FastSet<T>` to keep the change
  easy to apply and easy to revert.
- Use the alias only for internal engine state, not on public FFI-facing types.

**Reasoning**

- The default hasher optimizes for adversarial input resistance, not raw speed.
- Ferric's runtime state is dominated by internal keys generated by the engine,
  compiler, or parser, which makes a faster non-cryptographic hasher a better
  fit.
- This is the cleanest broad-spectrum experiment because it can be applied
  incrementally and rolled back easily.

**Risk**

- Low. Behavior should be unchanged.
- The main decision is whether deterministic (`Fx*`) or randomized (`A*`)
  hashing is preferable for the project.

### 2. Replace Tuple-Key Name Registries with Nested Maps

**Current structures**

- `crates/ferric-runtime/src/functions.rs` stores:
  - `FunctionEnv.functions: HashMap<(ModuleId, String), UserFunction>`
  - `GlobalStore.values: HashMap<(ModuleId, String), Value>`
  - `GenericRegistry.generics: HashMap<(ModuleId, String), GenericFunction>`
- `crates/ferric-runtime/src/engine.rs` also stores:
  - `function_modules: HashMap<(ModuleId, String), ModuleId>`
  - `global_modules: HashMap<(ModuleId, String), ModuleId>`
  - `generic_modules: HashMap<(ModuleId, String), ModuleId>`

**Proposed substitution**

- Replace the tuple-key tables with nested maps:
  - `FastMap<ModuleId, FastMap<Box<str>, T>>`
- If we prefer not to switch hashers first, the same layout still helps even
  with `std::HashMap`.

**Where we'd use it**

- All module/name registries in `functions.rs` and the matching module lookup
  tables in `engine.rs`.

**Additional adjustments**

- Remove helper paths that build temporary owned `String` keys for lookups.
- Refactor lookup helpers to take `&str` and perform two borrowed lookups:
  module first, then name.
- Update any "modules for name" scans to work against the nested structure.
- Store the string keys as `Box<str>` instead of `String`.

**Reasoning**

- These tables are hit on evaluator resolution paths.
- The current tuple layout forces hashing both the module ID and the owned
  string together.
- It also encourages temporary `String` creation during lookup, which adds
  allocation cost on a hot path.
- Splitting by module first and name second matches the actual lookup shape and
  reduces both hashing work and allocation pressure.

**Risk**

- Low to moderate.
- The structure is straightforward, but it touches several related registries,
  so this should land as one tightly scoped change with focused tests.

**Experiment note (2026-02-28)**

- `crates/ferric-runtime/src/functions.rs` had already moved to the nested
  `ModuleNameMap<T>` layout before this pass, so the remaining work here was the
  three module-ownership bookkeeping maps in `crates/ferric-runtime/src/engine.rs`.
- Converting `function_modules`, `global_modules`, and `generic_modules` from
  `HashMap<(ModuleId, String), ModuleId>` to `ModuleNameMap<ModuleId>` was kept.
- The targeted load-path microbenchmarks all stayed within Criterion's noise
  threshold after the conversion:
  - `function_owner_map_load_cycle`: roughly `310.7 us` to `314.7 us`
  - `global_owner_map_load_cycle`: roughly `228.0 us` to `226.4 us`
  - `generic_owner_map_load_cycle`: roughly `173.8 us` to `176.1 us`
- That makes the runtime impact effectively neutral while aligning these fields
  with the already-nested registries and removing the temporary owned-string
  lookup in debug consistency checks.

### 3. Replace `RuleId`-Keyed Hash Maps with Dense Indexed Vectors

**Current structures**

- `crates/ferric-runtime/src/engine.rs` stores:
  - `rule_info: HashMap<RuleId, Rc<CompiledRuleInfo>>`
  - `rule_modules: HashMap<RuleId, ModuleId>`

**Proposed substitution**

- Replace both with dense indexed storage:
  - `Vec<Option<Rc<CompiledRuleInfo>>>`
  - `Vec<Option<ModuleId>>`
- A small wrapper type around these vectors would keep indexing logic contained.

**Where we'd use it**

- Runtime rule metadata lookup in `Engine`.

**Additional adjustments**

- Grow the vectors when inserting a new rule.
- Index by `rule_id.0 as usize`.
- Keep accessor helpers so call sites do not depend on the underlying storage.
- Iteration over rules should skip empty slots.

**Reasoning**

- `RuleId` is monotonic and effectively dense.
- Hashing sequential integer IDs adds overhead without buying anything useful.
- Rule metadata is consulted during activation handling, so this removes a
  repeated hash-table round trip from a hot path.

**Risk**

- Low.
- The main care point is preserving behavior if rule IDs are ever sparse due to
  reset/reload behavior; using `Option` slots handles that safely.

**Experiment note (2026-02-28)**

- `crates/ferric-runtime/src/engine.rs` already uses `RuleIndex<T> = Vec<Option<T>>`
  for both `rule_info` and `rule_modules`.
- No additional runtime experiment was needed here; this candidate is already
  satisfied by the current implementation.

### 4. Make the Template Registry Fully `slotmap`-Native

**Current structures**

- `crates/ferric-runtime/src/engine.rs` stores:
  - `template_id_alloc: slotmap::SlotMap<TemplateId, ()>`
  - `template_defs: HashMap<TemplateId, RegisteredTemplate>`
  - `template_modules: HashMap<TemplateId, ModuleId>`

**Proposed substitution**

- Collapse this into:
  - `SlotMap<TemplateId, RegisteredTemplate>`
  - `SecondaryMap<TemplateId, ModuleId>`

**Where we'd use it**

- The template definition registry in `Engine`.

**Additional adjustments**

- Insert `RegisteredTemplate` directly into the `SlotMap` to obtain the
  `TemplateId`.
- Remove the separate `template_id_alloc`.
- Keep `template_ids: HashMap<String, TemplateId>` (or a refined variant of it)
  as the name-to-ID index.
- Update clear/reset code to reset the `SlotMap` and `SecondaryMap` together.

**Reasoning**

- `TemplateId` already exists specifically because a slotmap allocator is in use.
- The current arrangement allocates IDs with `SlotMap`, then immediately pays
  for a second hash lookup to reach the actual data.
- Using `SlotMap` for the data and `SecondaryMap` for related metadata removes
  duplicated indexing layers and better matches the existing ID model.

**Risk**

- Low.
- This is largely a structural simplification of storage that is already keyed
  by a `slotmap` ID.

**Experiment note (2026-02-28)**

- `crates/ferric-runtime/src/engine.rs` already stores template definitions in
  `SlotMap<TemplateId, RegisteredTemplate>` and template-module ownership in
  `SecondaryMap<TemplateId, ModuleId>`.
- No additional runtime experiment was needed here; this candidate is already
  satisfied by the current implementation.

### 5. Replace `slotmap`-Keyed Side Tables with `SecondaryMap` / `SparseSecondaryMap`

**Current structures**

- Several tables use `HashMap` keyed by `slotmap` IDs even though the primary
  stores are already `SlotMap`s.
- The strongest candidates are:
  - `crates/ferric-core/src/agenda.rs`
    - `id_to_key: HashMap<ActivationId, AgendaKey>`
    - `token_to_activations: HashMap<TokenId, SmallVec<[ActivationId; 2]>>`
  - `crates/ferric-core/src/token.rs`
    - `fact_to_tokens: HashMap<FactId, SmallVec<[TokenId; 4]>>`
    - `parent_to_children: HashMap<TokenId, SmallVec<[TokenId; 4]>>`
  - `crates/ferric-core/src/alpha.rs`
    - `fact_to_memories: HashMap<FactId, SmallVec<[AlphaMemoryId; 4]>>`

**Proposed substitution**

- Use `SecondaryMap` for dense 1:1 relationships.
- Use `SparseSecondaryMap` for sparse reverse indices and fan-out tables.

**Where we'd use it**

- Side indexes whose keys are `ActivationId`, `TokenId`, `FactId`, or other
  `slotmap` keys.

**Additional adjustments**

- Replace `HashMap`-specific `entry` patterns with the corresponding
  `SparseSecondaryMap` access patterns.
- Keep helper functions around insertion/removal so future storage changes stay
  localized.
- Validate clear/retract paths carefully because these structures are updated
  in lockstep with the owning stores.

**Reasoning**

- These keys are already generated by `slotmap`, so the system already has a
  direct indexing story available.
- Hashing them again for side tables is extra overhead.
- `SparseSecondaryMap` is especially attractive for reverse indices because it
  preserves the sparse nature of the table without forcing vector-sized storage.

**Risk**

- Low to moderate.
- The storage types are appropriate, but the affected code tends to sit in the
  assert/retract machinery, so test coverage matters.

**Experiment note (2026-02-28)**

- Converting `TokenStore.fact_to_tokens` and `TokenStore.parent_to_children`
  in `crates/ferric-core/src/token.rs` from `HashMap` to
  `SparseSecondaryMap` was tested and reverted.
- The targeted microbenchmark `token_store_reverse_index_cycle` regressed from
  roughly `60.1 us` to `109.8 us` (about `+80%`), so that specific
  substitution is not a good fit for this code path.
- The likely issue is `SparseSecondaryMap`'s insertion/update path for these
  fan-out lists being more expensive than the existing `FxHashMap` `entry`
  pattern under this workload.
- Converting `AlphaNetwork.fact_to_memories` in `crates/ferric-core/src/alpha.rs`
  from `HashMap` to `SparseSecondaryMap` was kept.
- The targeted microbenchmark `alpha_network_reverse_index_cycle` moved from
  roughly `63.4 us` to `64.0 us`, which Criterion flagged as within the noise
  threshold, so it meets the current "no less efficient" acceptance bar while
  simplifying the slotmap-keyed reverse index.
- Converting `Agenda.id_to_key` in `crates/ferric-core/src/agenda.rs` from
  `HashMap` to `SecondaryMap` was kept.
- The targeted microbenchmark `agenda_token_index_cycle` improved from roughly
  `103.4 us` to `101.0 us` (about `-2.5%`), making the dense slotmap side table
  a straightforward win for this field.
- Converting `Agenda.token_to_activations` in `crates/ferric-core/src/agenda.rs`
  from `HashMap` to `SparseSecondaryMap` was tested and reverted.
- Using a fresh post-`id_to_key` baseline, the targeted microbenchmark
  `agenda_token_index_cycle` regressed from roughly `99.7 us` to `106.8 us`
  (about `+7.4%`), so the sparse slotmap fan-out table is not a good fit for
  this path.

### 6. Convert Slotmap-Keyed Runtime State Maps in Negative/Exists/NCC Paths

**Current structures**

- `crates/ferric-core/src/negative.rs`
  - `blocked: HashMap<TokenId, HashSet<FactId>>`
  - `fact_to_blocked: HashMap<FactId, HashSet<TokenId>>`
  - `unblocked: HashMap<TokenId, TokenId>`
- `crates/ferric-core/src/exists.rs`
  - `support: HashMap<TokenId, HashSet<FactId>>`
  - `satisfied: HashMap<TokenId, TokenId>`
  - `fact_to_parents: HashMap<FactId, HashSet<TokenId>>`
- `crates/ferric-core/src/ncc.rs`
  - `result_count: HashMap<TokenId, usize>`
  - `result_owner: HashMap<TokenId, TokenId>`
  - `unblocked: HashMap<TokenId, TokenId>`
- `crates/ferric-core/src/beta.rs`
  - `BetaMemory.tokens: HashSet<TokenId>`

**Proposed substitution**

- First pass:
  - convert the outer `HashMap` layers to `SparseSecondaryMap`,
  - optionally convert `BetaMemory.tokens` to `SparseSecondaryMap<TokenId, ()>`
    or a fast-hash set if the API becomes awkward.
- Leave the inner `HashSet`s alone for the first experiment unless the change
  is bundled with the fast-hasher pass.

**Where we'd use it**

- The negative condition, exists, NCC, and beta-memory bookkeeping paths.

**Additional adjustments**

- Add small helper methods to preserve the current semantics around "present /
  not present" and count queries.
- Keep the outer-map conversion separate from any inner-set conversion so the
  measurements stay clean.

**Reasoning**

- These structures are part of the highest-churn state in the rete network.
- They are keyed almost entirely by slotmap IDs.
- Converting the outer layer removes one whole hash-table lookup per operation
  while keeping the inner membership logic unchanged.

**Risk**

- Moderate.
- The logic is sensitive, but the change can be phased so the first experiment
  remains mechanically simple.

**Experiment note (2026-02-28)**

- Converting the outer `HashMap` layers in `crates/ferric-core/src/negative.rs`
  to `SparseSecondaryMap` was tested and reverted.
- The targeted microbenchmark `negative_memory_outer_index_cycle` regressed
  from roughly `22.9 us` to `44.8 us` (about `+94%`).
- Converting the outer `HashMap` layers in `crates/ferric-core/src/exists.rs`
  to `SparseSecondaryMap` was tested and reverted.
- The targeted microbenchmark `exists_memory_outer_index_cycle` regressed from
  roughly `24.8 us` to `52.1 us` (about `+111%`).
- Converting the outer `HashMap` layers in `crates/ferric-core/src/ncc.rs`
  to `SparseSecondaryMap` was tested and reverted.
- The targeted microbenchmark `ncc_memory_outer_index_cycle` regressed from
  roughly `7.24 us` to `24.3 us` (about `+236%`).
- Across all three, the current high-churn update patterns favor the existing
  `FxHashMap`-based outer maps over `SparseSecondaryMap`.
- Converting `BetaMemory.tokens` in `crates/ferric-core/src/beta.rs` from
  `HashSet<TokenId>` to `SparseSecondaryMap<TokenId, ()>` was also tested and
  reverted.
- Reusing the existing `beta_memory_store_cycle` microbenchmark, the change
  regressed from roughly `101.6 us` to `339.4 us` (about `+236%`), so the
  token-membership path strongly favors the current `FxHashSet`.

### 7. Replace Dense `u32`-Indexed Maps with `Vec` / `Vec<Option<_>>` in Core Networks

**Current structures**

- `crates/ferric-core/src/alpha.rs`
  - `nodes: HashMap<NodeId, AlphaNode>`
  - `memories: HashMap<AlphaMemoryId, AlphaMemory>`
- `crates/ferric-core/src/beta.rs`
  - `memories: HashMap<BetaMemoryId, BetaMemory>`
  - `neg_memories: HashMap<NegativeMemoryId, NegativeMemory>`
  - `ncc_memories: HashMap<NccMemoryId, NccMemory>`
  - `exists_memories: HashMap<ExistsMemoryId, ExistsMemory>`

**Proposed substitution**

- Replace these with `Vec<_>` or `Vec<Option<_>>`, indexed directly by the ID's
  numeric payload.

**Where we'd use it**

- Dense append-only stores whose IDs are allocated sequentially.

**Additional adjustments**

- Keep `get` / `get_mut` helpers so callers do not depend directly on vector
  indexing.
- Use `Option` slots if there is any chance of gaps after resets or removals.
- Do **not** apply this blindly to `BetaNetwork.nodes`: that table is seeded at
  a high `NodeId` value and is intentionally sparse, so a vector would waste
  space there.

**Reasoning**

- For dense sequential IDs, a hash map is just paying for indirection and
  hashing that direct indexing avoids.
- This should reduce both CPU cost and per-entry overhead on core network state.

**Risk**

- Low to moderate.
- Safe where density is real; harmful where IDs are intentionally sparse.

**Experiment note (2026-02-28)**

- Converting `AlphaNetwork.memories` in `crates/ferric-core/src/alpha.rs`
  from `HashMap<AlphaMemoryId, AlphaMemory>` to `Vec<AlphaMemory>` was kept.
- The targeted microbenchmark `alpha_network_reverse_index_cycle` improved from
  roughly `70.4 us` to `65.3 us` (about `-7.2%`), which makes this a clear win
  for one of the dense sequential-ID stores in `AlphaNetwork`.
- Converting `AlphaNetwork.nodes` in `crates/ferric-core/src/alpha.rs` from
  `HashMap<NodeId, AlphaNode>` to `Vec<AlphaNode>` was kept.
- The targeted microbenchmark `alpha_node_store_cycle` improved from roughly
  `1.72 ms` to `1.61 ms` (about `-5.5%`), making direct indexing a clear win
  for the dense alpha-node table as well.
- Converting `BetaNetwork.memories` in `crates/ferric-core/src/beta.rs` from
  `HashMap<BetaMemoryId, BetaMemory>` to `Vec<BetaMemory>` was kept.
- The targeted microbenchmark `beta_memory_store_cycle` improved from roughly
  `202.3 us` to `101.1 us` (about `-50%`), making direct indexing a clear win
  for the main beta-memory table as well.
- Converting `BetaNetwork.neg_memories` in `crates/ferric-core/src/beta.rs`
  from `HashMap<NegativeMemoryId, NegativeMemory>` to `Vec<NegativeMemory>` was
  kept.
- The targeted microbenchmark `beta_negative_memory_store_cycle` improved from
  roughly `2.12 ms` to `2.00 ms` (about `-5.6%`), so the dense negative-memory
  table also benefits from direct indexing.
- Converting `BetaNetwork.ncc_memories` in `crates/ferric-core/src/beta.rs`
  from `HashMap<NccMemoryId, NccMemory>` to `Vec<NccMemory>` was kept.
- The targeted microbenchmark `beta_ncc_memory_store_cycle` improved from
  roughly `654 us` to `499 us` (about `-24%`), making direct indexing another
  clear win for the dense NCC-memory table.
- Converting `BetaNetwork.exists_memories` in `crates/ferric-core/src/beta.rs`
  from `HashMap<ExistsMemoryId, ExistsMemory>` to `Vec<ExistsMemory>` was kept.
- The targeted microbenchmark `beta_exists_memory_store_cycle` improved from
  roughly `2.24 ms` to `2.12 ms` (about `-5.5%`), so the dense exists-memory
  table also benefits from direct indexing.

### 8. Replace Tiny Hash Sets with `SmallVec` in Clearly Small Cardinality Cases

**Current structures**

- `crates/ferric-core/src/alpha.rs`
  - `AlphaMemory.indexed_slots: HashSet<SlotIndex>`

**Proposed substitution**

- Replace with `SmallVec<[SlotIndex; 4]>`.

**Where we'd use it**

- Only for clearly bounded, small-cardinality sets where membership checks are
  rare enough that linear scan is cheaper than hashing.

**Additional adjustments**

- Replace `insert` with a short dedupe scan before `push`.
- Replace `contains` with a linear membership check.
- Keep the change local to this field; do not apply the same pattern broadly
  until measured.

**Reasoning**

- `indexed_slots` should usually contain only the small number of slots that are
  actually indexed for a given alpha memory.
- In that shape, the hash set's table allocation and hashing overhead are
  likely more expensive than a short inline array scan.
- `smallvec` is already a workspace dependency, so there is no new dependency
  cost here.

**Risk**

- Low.
- This is a classic "small set" optimization and easy to revert.

**Experiment note (2026-02-28)**

- Converting `AlphaMemory.indexed_slots` in `crates/ferric-core/src/alpha.rs`
  from `HashSet<SlotIndex>` to `SmallVec<[SlotIndex; 4]>` was kept.
- The targeted microbenchmark `alpha_memory_indexed_slots_cycle` improved from
  roughly `37.2 us` to `35.5 us` (about `-4.5%`), confirming that the small
  inline set fits this access pattern better than hashing.

### 9. Replace Tiny Fan-Out Lists with `SmallVec`

**Current structures**

- `crates/ferric-core/src/beta.rs`
  - `alpha_to_joins: HashMap<AlphaMemoryId, Vec<NodeId>>`
  - `alpha_to_negatives: HashMap<AlphaMemoryId, Vec<NodeId>>`
  - `alpha_to_exists: HashMap<AlphaMemoryId, Vec<NodeId>>`
- There are also temporary borrow-breaking scratch copies in propagation paths
  that currently materialize heap `Vec`s.

**Proposed substitution**

- Replace the `Vec<NodeId>` values with `SmallVec<[NodeId; 2]>` or
  `SmallVec<[NodeId; 4]>`.
- Replace scratch `Vec` clones in propagation planning with `SmallVec` as well.

**Where we'd use it**

- Small subscriber lists and transient copies in rete propagation.

**Additional adjustments**

- Choose the inline capacity based on observed fan-out (start conservative).
- Keep the outer map shape unchanged for the first experiment so only the value
  storage changes.

**Reasoning**

- Most alpha memories are unlikely to fan out to many beta nodes.
- When the common case is one or two subscribers, heap-allocating a `Vec` for
  every stored adjacency list is unnecessary overhead.
- This is a good "cheap allocation reduction" experiment because the semantic
  model does not change at all.

**Risk**

- Low.
- The main tuning decision is choosing the inline capacity.

**Experiment note (2026-02-28)**

- Converting the beta subscriber lists in `crates/ferric-core/src/beta.rs`
  (`alpha_to_joins`, `alpha_to_negatives`, `alpha_to_exists`) from
  `Vec<NodeId>` to `SmallVec<[NodeId; 4]>` was kept.
- The matching borrowed-clone scratch lists in `crates/ferric-core/src/rete.rs`
  were also switched from heap `Vec` clones to `SmallVec::from_slice(...)`.
- The targeted microbenchmark `beta_fanout_index_cycle` improved from roughly
  `907 us` to `154 us` (about `-83%`) on the small-fan-out workload it was
  designed to represent, which makes this a clear win for the intended case.

### 10. Tighten Long-Lived String-Key Storage with `Box<str>` and Small-Map Options

**Current structures**

- `crates/ferric-runtime/src/templates.rs`
  - `RegisteredTemplate.slot_index: HashMap<String, usize>`
- `crates/ferric-runtime/src/modules.rs`
  - `name_to_id: HashMap<String, ModuleId>`
- `crates/ferric-runtime/src/engine.rs`
  - `template_ids: HashMap<String, TemplateId>`

**Proposed substitution**

- Minimum-risk pass:
  - change owned keys from `String` to `Box<str>`,
  - pair that with a faster hash backend.
- Higher-upside follow-up for very small maps:
  - replace `RegisteredTemplate.slot_index` with
    `SmallVec<[(Box<str>, usize); 8]>` plus linear lookup.

**Where we'd use it**

- Long-lived registries whose keys are inserted once and then read many times.
- Per-template slot lookup is the best candidate for the small-map variant
  because templates typically have modest slot counts.

**Additional adjustments**

- Centralize lookup behind helper methods so callers can still use `&str`.
- Apply the `Box<str>` conversion first, then evaluate whether the `SmallVec`
  variant is actually better for `slot_index`.

**Reasoning**

- `String` stores length, capacity, and pointer; `Box<str>` removes spare
  capacity and trims per-entry memory for immutable stored keys.
- These registries are mostly read-only after construction, so a more compact
  owned string representation makes sense.
- `RegisteredTemplate.slot_index` is especially likely to be small enough that
  a tiny linear structure can outperform a hash map.

**Risk**

- Low for `Box<str>`.
- Moderate for the `SmallVec` small-map variant, because it changes the lookup
  code shape and depends on actual template slot counts.

**Experiment note (2026-02-28)**

- Converting `RegisteredTemplate.slot_index` in
  `crates/ferric-runtime/src/templates.rs` from `HashMap<String, usize>` to
  `HashMap<Box<str>, usize>` was tested and reverted.
- In `template_registry_bench`, `load_many_templates` showed no meaningful
  change, but `template_modify_reset_run` regressed from roughly `1.75 us` to
  `1.84 us` (about `+4.1%`), so this specific `Box<str>` substitution is not a
  win for the current slot-lookup path.
- Converting `ModuleRegistry.name_to_id` in
  `crates/ferric-runtime/src/modules.rs` from `HashMap<String, ModuleId>` to
  `HashMap<Box<str>, ModuleId>` was kept.
- The targeted microbenchmark `module_registry_lookup_cycle` shifted from
  roughly `16.54 us` to `16.78 us`; Criterion reported that as within the
  configured noise threshold, so the runtime cost is effectively flat while the
  stored keys become more compact.
- Converting `Engine.template_ids` in `crates/ferric-runtime/src/engine.rs`
  from `HashMap<String, TemplateId>` to `HashMap<Box<str>, TemplateId>` was

### 14. Replace `VarMap`'s Symbol Hash Lookup with Direct Pool Indexing

**Current structures**

- `crates/ferric-core/src/binding.rs`
  - `VarMap.by_name: HashMap<Symbol, VarId>`
  - `VarMap.by_id: Vec<Symbol>`

**Proposed substitution**

- Replace the hash map with two direct-indexed vectors:
  - `ascii_by_name: Vec<Option<VarId>>`
  - `utf8_by_name: Vec<Option<VarId>>`

**Where we'd use it**

- Variable-name to `VarId` resolution inside `VarMap`.

**Additional adjustments**

- Keep `by_id: Vec<Symbol>` unchanged so reverse lookup stays identical.
- Dispatch on the symbol pool (`ASCII` vs `UTF-8`) and index directly by the
  interned symbol ID within that pool.
- Add a mixed-pool microbenchmark that includes padded symbol IDs before the
  variable names, so the measurement includes nontrivial vector growth and
  both symbol pools.

**Reasoning**

- `Symbol` values are already interned and carry compact numeric IDs.
- Re-hashing them in `VarMap` adds avoidable overhead on a path that is hit
  repeatedly during pattern compilation.
- Splitting the storage by symbol pool preserves the existing semantics while
  turning lookup into a bounds check plus direct indexing.

**Risk**

- Low to moderate.
- CPU cost should improve, but memory use can grow if variable symbols arrive
  with very sparse symbol IDs. That tradeoff needs to stay visible.

**Experiment note (2026-02-28)**

- Added a dedicated `var_map_symbol_lookup_cycle` microbenchmark to
  `crates/ferric-core/benches/storage_indices_bench.rs`.
- Converting `VarMap.by_name` in `crates/ferric-core/src/binding.rs` from
  `HashMap<Symbol, VarId>` to split direct-indexed vectors
  (`ascii_by_name` / `utf8_by_name`) was kept.
- Using the new targeted benchmark, the mixed-pool insert/lookup cycle
  improved from roughly `5.96 us` to `2.42 us` (about `-59.3%`), making this
  a clear win for the current symbol-ID-driven lookup path.

### 15. Replace `FactBase.by_relation`'s Symbol Hash Lookup with Direct Pool Indexing

**Current structures**

- `crates/ferric-core/src/fact.rs`
  - `FactBase.by_relation: HashMap<Symbol, HashSet<FactId>>`

**Proposed substitution**

- Replace the outer symbol-keyed hash map with split direct-indexed vectors:
  - `ascii: Vec<Option<HashSet<FactId>>>`
  - `utf8: Vec<Option<HashSet<FactId>>>`

**Where we'd use it**

- Ordered-fact relation lookup in `FactBase`.

**Additional adjustments**

- Keep the inner `HashSet<FactId>` buckets unchanged so the experiment isolates
  only the outer symbol-key lookup.
- Add a dedicated mixed-pool benchmark with padded symbol IDs to measure a
  less-dense symbol-ID workload alongside the existing dense relation-index
  benchmark.
- Preserve the existing `facts_by_relation` API and index cleanup behavior.

**Reasoning**

- Relation names are already interned `Symbol` values with compact numeric IDs.
- The existing structure hashes those IDs on every insert, lookup, and retract.
- Direct pool indexing keeps the same semantics while reducing the outer-key
  path to bounds checks and vector access.

**Risk**

- Low to moderate.
- CPU cost should improve, but memory use can grow if relation symbols are very
  sparse within a symbol table.

## Areas to Deprioritize for Now

These are existing structures that currently look well-matched to their
workload and are not obvious "slot-in replacement" candidates:

- `crates/ferric-core/src/agenda.rs`
  - `ordering: BTreeMap<AgendaKey, ActivationId>`
  - This matches the need for ordered retrieval, exact-key removal, and
    forward scans. Changing it would be more invasive than the other items.

- `crates/ferric-runtime/src/engine.rs`
  - `input_buffer: VecDeque<String>`
  - The workload is straightforward FIFO `push_back` / `pop_front`, which is
    exactly what `VecDeque` already optimizes for.

## Suggested Experiment Order

If the goal is to maximize odds of finding early, clean wins:

1. Fast-hash alias pass for internal maps/sets.
2. Tuple-key registry split into nested maps.
3. `RuleId` vectorization.
4. Template registry `SlotMap` / `SecondaryMap` cleanup.
5. `slotmap` side-table conversions (`SecondaryMap` / `SparseSecondaryMap`).
6. Small-collection passes (`SmallVec` for `indexed_slots`, fan-out lists, and
   possibly `slot_index`).

That ordering starts with the least invasive, most broadly applicable changes,
then moves toward more targeted structure-specific tuning.
